ARE THE THINGS WE SHIP ACTUALLY BEING USED?

   

Problem

Product teams need to build products that are valuable to users.

But how do you know that what you build is valuable?

One way to get closer to an answer is to assess product usage. Product usage can be thought of as a signal for demand for your product. Working as the the user researcher with Wrap Media, I recently set out to measure product usage.

Solution

At Wrap Media, all our event data was collected by FullStory, and available through their Rest API. As a team, we got together, reviewed the data available to us, and came up with a plan to measure product useage.

  • Actionability - product usege data would be used to prioritize which existing featues should be iteratd upon, and which new features should be added to the roadmap.

  • Clear definition - usage would be defind as a click event on a particular feature - for example clicking export on our data export button.

  • Granularity - usage would be reported on a weekly basis, starting in the week the feature was available to a general audience.

  • Customer context - the data would be segmented by customer to understand who is (not) using a feature.

With the entire team aligned on the goals and requirements for the measurement, it was time for me to extract, prepare, and visualize the data.

 

Extract

I used Hadley Wickham's httr package to extract data bundles from the FullStory REST API.

ds <- (bind_rows(
  lapply(
    list.files(
      "file_path", 
      pattern = "bundle.gz", 
      full.names = TRUE

    fromJSON
  )
)) %>%
  arrange(desc(EventStart))

Prepare

Next, I excluded internal users so that a QA engineer testing a feature for example would not bias the feature useage measurement.

ext_only <- filter(
    ds,
    !(grepl("wrap.co", UserEmail) & UserEmail != "design@wrap.co") 
    & !grepl("argodesign.com|mauricio.andrada@gmail.com",
    UserEmail)
    )

Then I created the different segments for our main customers.

customer <- ext_only %>%
    mutate(customer = case_when(
        (grepl("customer_A_email", ext_only$UserEmail)) ~ "Customer A",
        (grepl("customer_B_email", ext_only$UserEmail)) ~ "Customer B",
        (grepl("customer_C_email", ext_only$UserEmail)) ~ "Customer C",
        )

Visualize

The data was then visualized using ggplot2, and displayed in a dashboard.

Right away, you can see that Feautes A and D see quite a bit more usage than Featues B and C. Visualizations like this can bring a lot of insight to a product team. For example, Product Managers at Wrap Media no longer had to guess which features in the product were being used more than others.

Product acted on these insights to priortitize roadmap planning and product iterations.
